## 1. システム概要

### 1.1 アーキテクチャ概要

```
[エンドユーザー]
    │ LINEアプリ
    ▼
[LIFF アプリ]  ←── GitHub Pages ホスティング
    │ liff.html / liff.js / liff.css
    │ URL: ?gasApiUrl=<GAS_URL>&liffId=<LIFF_ID>
    │
    │ HTTPS POST (JSON + idToken)
    ▼
[GAS Web App]  ←── Google Apps Script デプロイ
    │ Code.js / calendar-addon.js / settings.html / kanri.html / unauthorized.html
    │ executeAs: USER_DEPLOYING / access: ANYONE_ANONYMOUS
    │
    ├── Google Sheets（データベース）
    ├── Google Calendar（予約イベント・ブロック予定）
    └── LINE Messaging API（Webhook受信・プッシュ通知）

[店舗スタッフ（PC）]
    │ Googleアカウントでアクセス
    ├── Googleカレンダー → カレンダーアドオン（来店確認・キャンセル）
    └── GAS Web App doGet
            ├── ?page=kanri  → kanri.html（予約管理画面）
            └── デフォルト   → settings.html（システム設定画面）

[店舗スタッフ（スマホ）]
    │ ブラウザでシステムURLをブックマーク
    └── kanri.html（予約管理画面）

[新規店舗オーナー]
    │
    ▼
[Cloudflare Worker]  ←── onboarding + デプロイ支援コンソール
    │ Google OAuth 2.0（オンボーディング）
    │ /admin: Cloudflare Access + Cookie セッション（開発者のみ）
    └── 自動プロビジョニング / コード配信 / 顧客台帳
```

### 1.2 ホスティング構成

| コンポーネント | ホスティング先 | URL形式 |
|---|---|---|
| LIFFアプリ（予約画面） | GitHub Pages | `https://[org].github.io/[repo]/liff.html` |
| GAS バックエンド | Google Apps Script | `https://script.google.com/macros/s/[ID]/exec` |
| オンボーディングサーバー | Cloudflare Workers | `https://[name].[subdomain].workers.dev` |

## 3. バックエンド API設計（Code.js）

### 3.1 doGet エンドポイント

```
GET [GAS_URL]
GET [GAS_URL]?page=kanri
GET [GAS_URL]?action=downloadICS&bookingIds=BK001,BK002
```

**認証**: `Session.getActiveUser().getEmail()` を `adminEmail` および `staffs[].email` と照合。未認証時は `unauthorized.html` を返す。

**ICSダウンロード**: 認証不要（URLを知っている全員がアクセス可能）。指定された予約IDのICSファイルを返す。VALARM（1時間前通知）を含む。

### 3.2 doPost エンドポイント（LIFF API）

**認証（v3.1）**: 予約系 action は `idToken`（LIFF `getIDToken()`）を GAS が LINE Verify API で検証し、リクエストの `lineUserId` と一致することを要求。詳細は [セキュリティ設計書.md §6](./セキュリティ設計書.md)。

**リクエスト形式**:
```json
{"action": "アクション名", "idToken": "...", "lineUserId": "U...", ...パラメーター}
```

**レスポンス形式**:
```json
{"success": true, "data": {}}
{"success": false, "message": "エラー内容"}
{"success": false, "code": "LINE_AUTH_FAILED", "message": "..."}
```

#### アクション一覧

| アクション | 引数 | ID Token | 説明 |
|---|---|---|---|
| `getInitData` | `lineUserId`, `idToken` | ✅ | 初期データ取得 |
| `getAvailableSlots` | `date`, `duration`, `staffEmail?` | — | 週間空き枠取得 |
| `createBooking` | `bookingData`, `idToken` | ✅ | 単一予約作成 |
| `createBulkBookings` | `bookingDataList`, `idToken` | ✅ | 一括予約作成 |
| `registerCustomer` | `lineUserId`, `customerName`, `phone?`, `idToken` | ✅ | 顧客登録 |

#### getInitData レスポンス

```json
{
  "shopName": "string",
  "serviceMenus": [{"name": "string", "duration": number}],
  "isStaffFeatureEnabled": boolean,
  "staffs": [{"name": "string", "email": "string"}],
  "bookingTimeUnit": number,
  "bookingLookaheadDays": number,
  "isRegistered": boolean,
  "maxBulkBookings": number
}
```

#### getAvailableSlots レスポンス

```json
{
  "yyyy-MM-dd": ["HH:mm", "HH:mm", ...],
  ...
}
```

**空き枠計算ロジック（重要）**:
- **Aルート（指名なし）**: 満席タイムテーブルから高速取得 → maxConcurrentBookings と比較
- **Bルート（指名あり）**: 予約シートから該当担当者の予約をリアルタイム計算 → 上限=1
- ブロックカレンダー同期後に `ブロック予定同期` シートを参照してブロックスロットを除外
- **定休日（v3.2）**: `holidays` に含まれる曜日は当日の空き枠をすべて除外（任意設定）
- **非稼働時間（v3.2）**: `nonOperatingHours` の時間帯に該当するスロットを除外（任意設定。昼休み等）
- **当日予約ルール**: `sameDayMinHoursBefore` / `sameDayBlockedUntilTime` による当日枠の制限

#### createBooking / createBulkBookings

**一括予約の特記事項**:
- `LockService.getScriptLock()` で排他制御（タイムアウト30秒）
- 全スロットを先行チェック（Step1）→ 全件OK後に書き込み（Step2）
- バッチ内の予約同士の重複も検証
- 1件でも空きがなければ全件キャンセル（アトミック）

### 3.3 LINE Webhook

`doPost` で `postData.events` が存在する場合にWebhook処理。

| ユーザー送信テキスト | 応答内容 |
|---|---|
| `予約` | 今後の予約一覧（日時・メニュー・担当者） |

### 3.4 バックグラウンド処理（トリガー）

| 関数名 | 実行タイミング | 処理内容 |
|---|---|---|
| `runNightlyBatch` | 毎日 2:00 | 満席タイムテーブル全件再計算 |
| `syncBlockCalendars` | 毎時 | ブロックカレンダー差分同期（Sync Token方式） |
| `createEndOfDayUnconfirmedEvent` | 毎日 21:00 | 未確定予約がある場合カレンダーに警告予定を作成 |
| `sendDayBeforeReminders` | 毎日 18:00 | 翌日予約の顧客にLINEプッシュ通知（reminderMode=LINEのみ） |
| `sendHourBeforeReminders` | 毎時 | 60〜75分後の予約顧客にLINEプッシュ通知（reminderMode=LINEのみ） |
| `runDailyArchive` | 毎日 3:00 | 完了予約・無効顧客をアーカイブへ移動、古いタイムテーブル削除 |

## 4. フロントエンド設計（LIFF）

### 4.1 画面遷移

```
[起動] LIFF初期化 → LINE認証 → getInitData
    │
    ▼
Step1: 予約経験選択
    ├── 初めて + 未登録 → 顧客情報登録画面
    │       └── 登録完了 → Step2
    └── 2回目以降 / 既登録 → Step2
    │
    ▼
Step2: 予約種別選択（新規予約 / 確認変更※未実装）
    │
    ▼
Step3: メニュー選択
    │
    ▼
Step4: 担当者選択 → カレンダー表示 → 日時選択
    │ ※担当者未選択時はカレンダー非表示
    │
    ▼
予約確定ボタン → createBooking / createBulkBookings
    │
    ▼
完了画面（ICSダウンロード / Googleカレンダー追加）
```

### 4.2 URL パラメーター

LIFFアプリのURLには以下のクエリパラメーターが必要。sessionStorageにキャッシュされる。

```
?gasApiUrl=[GAS WebアプリURL]&liffId=[LIFF ID]
```

**認証**: URL を知っていても LINE ログイン（`liff.login()`）が必要。API 呼び出し時は `idToken` を付与し GAS で検証（`lineLoginChannelId` 設定時）。リッチメニュー経由でなくとも LIFF URL から予約可能なため、URL の公開範囲は最小限にすること。

### 4.3 一括予約UI仕様

- `maxBulkBookings` が 1 の場合: 一括選択UI非表示（単一予約のみ）
- `maxBulkBookings` が 2以上の場合: カウンターチップ表示
  - 未選択時: 「カレンダーから日時を選んでください（最大N件）」
  - 選択中: 「選択済み: X / N 件（タップで確認）▼」
  - 上限到達: 未選択スロットを `bulk-limit-reached`（グレーアウト）
  - 時間帯重複: `slot-conflict`（警告色）で選択不可

### 4.4 ICSダウンロード

- GASエンドポイント `?action=downloadICS&bookingIds=...` を呼び出す
- iOS LIFF環境では `liff.openWindow({ external: true })` でSafariを開く
- ICSには VALARM（1時間前通知）を含む

## 5. 管理画面設計

### 5.1 認証

`Session.getActiveUser().getEmail()` を以下と照合（大文字小文字無視）:
- `adminEmail`
- `staffs[].email`

**初回セットアップモード（v3.0追加）**:
- `adminEmail` が未設定の場合（オンボーディング直後など）、認証チェックをスキップしてアクセスを許可する
- `doGet`（画面表示）と `saveConfigs`（設定保存）の両方に適用
- `adminEmail` が設定済みの場合は通常の認証チェックに戻る
- これによりオンボーディング後に設定画面への初回アクセスが確実に可能となる

**saveConfigs（v3.2変更）**:
- Config シート A列に存在するキーは B列を更新
- **存在しないキー（例: `lineLoginChannelId`, `nonOperatingHours`）は新規行として追加**（v3.2）
- settings.html から保存した値が Config シートに確実に反映される

### 5.2 UIデザインシステム（v3.0追加）

全管理画面（settings.html / kanri.html）およびLIFFアプリ（liff.css）に共通のデザインシステムを適用。

| 要素 | 仕様 |
|---|---|
| フォント | Noto Sans JP（Google Fonts）|
| テーマ | Fresh Natural（温かみのあるナチュラル系）|
| 背景色 | `#F6F4F1`（温かみのあるオフホワイト）|
| アクセント色 | `#6B9E78`（セージグリーン）|
| テキスト色 | `#2A2A2A`（ウォームブラック）|
| カード影 | `0 2px 14px rgba(0,0,0,0.06)` |
| ボーダー半径 | カード: 14px、入力: 10px、ボタン: 10px |
| ボタン | グラデーション＋ホバーで浮き上がりエフェクト |

### 5.3 kanri.html（予約管理画面）

`[GAS_URL]?page=kanri` でアクセス。

**タブ構成**:

| タブ | 機能 |
|---|---|
| 予約一覧 | フィルター（本日/直近3日/全期間）、ステータス変更 |
| 終業後処理 | 本日の未確定予約一括確定（来店/キャンセル） |
| 顧客管理 | 顧客検索・編集（名前/電話/メモ/ステータス） |
| エラーログ | エラー一覧表示・クリア |

**ステータス変更時の連動処理**:
- `来店`: Googleカレンダーイベントを緑（Basil）に変更 + 顧客マスタの最終来店日更新
- `無断キャンセル`: Googleカレンダーイベントを赤（Tomato）に変更
- `キャンセル`: Googleカレンダーイベントを削除

### 5.4 settings.html（システム設定画面）

`[GAS_URL]` でアクセス。ヘッダー表示: **UI v3.2（定休日・非稼働時間対応）**

**設定セクション**:
1. 店舗基本情報（shopName, businessHours, bookingTimeUnit, bookingLookaheadDays, maxConcurrentBookings）
2. **定休日・非稼働時間（任意）**（holidays, nonOperatingHours）← v3.2追加
3. Googleカレンダー設定（reservationCalendarId, block_input_calendar_id）
4. 当日予約ルール（sameDayBlockedUntilTime, sameDayMinHoursBefore, alternateBookingGuide）
5. 複数日程一括予約設定（maxBulkBookings）
6. 担当者機能（isStaffFeatureEnabled, staffs, staff_block_calendar_id_map）
7. 提供メニュー（serviceMenus）
8. リマインド設定（reminderMode: ICS/LINE）
9. LINE連携設定（lineChannelAccessToken, lineChannelSecret, lineLoginChannelId, liffChannelId, Webhook URL / LIFF エンドポイントURL 表示）

**Config データの埋め込み（v3.1〜）**:
- `doGet` で `getConfigs()` の結果を HTML テンプレートに埋め込み
- GAS テンプレートは `<?!= ... ?>`（エスケープなし）を使用。`<?=` は JSON 破損の原因となるため使用しない

**LINE Login チャネルID**:
- Config キー: `lineLoginChannelId`
- 取得場所: LINE Developers › LINEログイン › チャネル基本設定 › チャネルID（数値）
- LIFF ID（`liffChannelId`）とは別物

## 6. ブロックカレンダー同期設計

### 6.1 同期戦略（Sync Token方式）

1. 初回（Sync Token なし）: フル同期（今日以降の全イベント）
2. 以降（Sync Token あり）: 差分同期（変更分のみ）
3. Sync Token 失効（410 Gone）: 自動でフル同期に切り替え

### 6.2 カレンダー選択ロジック

| 条件 | 使用カレンダー |
|---|---|
| 担当者機能OFF | `block_input_calendar_id`（店舗共有） |
| 担当者機能ON + 指名あり | `staff_block_calendar_id_map[staffEmail]` |
| 担当者機能ON + 指名なし | `block_input_calendar_id`（店舗共有） |

### 6.3 終日イベントの処理

終日イベントは該当日の営業時間全スロットをブロック。

## 7. データアーカイブ設計

### 7.1 アーカイブ対象

| シート | 条件 |
|---|---|
| 予約 | ステータスが「来店/キャンセル/無断キャンセル」かつ予約日が昨日以前 |
| 顧客マスタ | ステータスが「無効」 |
| 満席タイムテーブル | 日付が昨日以前（削除のみ、アーカイブ不要） |

### 7.2 アーカイブファイル管理

- 総行数が50,000行を超えた場合: 新しいアーカイブスプレッドシートを自動作成
- Config の `archiveSpreadsheetId` を自動更新
- ファイル名: `予約アーカイブ_[shopName]_[yyyy-MM]`

## 8. オンボーディング設計（Cloudflare Worker）

### 8.1 処理フロー

```
GET /           → ランディングページ
GET /start      → Google OAuth 開始（state をKVに保存）
GET /callback   → OAuth認証完了 → プロビジョニング実行
GET /complete   → 完了ページ（URL表示）
GET /admin/login  → 管理パスワード入力（Cookie 発行）
GET /admin        → デプロイ支援コンソール
```

### 8.2 プロビジョニング手順

1. OAuthトークンから店舗オーナーのメールアドレスを取得（`id_token` JWTデコード → 失敗時は `userinfo` API）
2. テンプレートスプレッドシートをコピー（`Drive API files.copy`）
3. ConfigシートのreservationSheetIdに新SSのIDを書き込み（`Sheets API`）
4. **ConfigシートのadminEmailに店舗オーナーのメールを書き込み**（`Sheets API`）← v3.0追加
5. 新規スタンドアロンGASプロジェクト作成（`Apps Script API projects.create`）
6. GitHubからスクリプトファイルを取得し、`PLACEHOLDER_SPREADSHEET_ID` を実IDに置換
7. GASプロジェクトへコンテンツをプッシュ（`projects.updateContent`）
8. バージョン作成（`projects.versions.create`）
9. Webアプリデプロイ（`projects.deployments.create`）

**ヘルパー関数**:
- `getEmailFromIdToken(idToken)` : JWT のペイロードをデコードしてメール取得（主）
- `getUserEmail(accessToken)` : userinfo API でメール取得（フォールバック）
- `writeConfigValue(ssId, auth, key, value)` : Configシートの任意キーへ値を書き込む汎用関数

### 8.3 OAuthスコープ（v3.0変更）

| スコープ | 変更 | 用途 |
|---|---|---|
| `drive.file` | **drive から変更**（v3.0） | テンプレートSSコピー（アプリ作成ファイルのみ）|
| `spreadsheets` | 変更なし | Configシートへの読み書き |
| `script.projects` | 変更なし | GASプロジェクト作成 |
| `script.deployments` | 変更なし | Webアプリデプロイ |
| `email` / `profile` | 変更なし | ユーザーメールアドレス取得 |

> **変更理由**: `drive`（全ドライブアクセス）はGoogleのCASA審査（有料）が必要。
> `drive.file`（アプリ作成ファイルのみ）に限定することでCASA不要・標準審査（無料）のみで対応可能。

|---|---|---|

### 8.6 デプロイ支援コンソール（/admin）

**認証（v3.1）**: Cloudflare Access（Gmail）+ HttpOnly Cookie セッション（`ADMIN_SECRET_KEY`）。旧 `?key=` URL は廃止。

**コード配信フロー**:
1. 開発者がローカルで修正 → `git add / commit / push origin main`（GitHub が正本）
2. LIFF（liff.js 等）は GitHub Pages に自動反映（1〜2分）
3. デプロイ支援コンソールで対象店舗を選択 →「コード配信」
4. Worker が `raw.githubusercontent.com` から GAS ソースを取得し、各店舗 GAS へ push

> **注意**: コンソールは GitHub から**読み取るのみ**。ローカル未 push の変更は配信されない。
> コンソール左パネルに git コマンド（コピー用）を表示（v3.2）。

**Worker 自体の更新**: `cloudflare-worker/` 変更時は `npx wrangler deploy` が別途必要。

## 9. カレンダーアドオン設計（v3.0追加）

### 9.1 概要

Googleカレンダーのサイドパネルに表示されるアドオン。カレンダーイベントを開いたときに、対応する予約情報を表示し、ステータス変更（来店確認・キャンセル）をその場で実行できる。

- **ファイル**: `calendar-addon.js`
- **デプロイ形式**: Google Workspace アドオン（テストデプロイ）
- **対象ユーザー**: 店舗スタッフ（PC Googleカレンダー）

### 9.2 エントリーポイント

| 関数 | トリガー | 説明 |
|---|---|---|
| `onCalendarHome(e)` | アドオンホーム表示時 | 使い方案内カードを表示 |
| `onCalendarEventOpen(e)` | カレンダーイベントを開いたとき | 予約情報カードを表示 |

> **注意**: アドオンはイベントを「クリック」しただけでは起動しない。タイトルをクリックして詳細画面を開いた場合のみ `onCalendarEventOpen` が呼ばれる（Google API仕様）。

### 9.3 予約情報取得フロー

```
カレンダーイベントを開く
    │
    ▼
onCalendarEventOpen(e)
    │ e.calendar.calendarEventId を取得
    ▼
findBookingByEventId_(calendarEventId)
    │ 予約シートの「イベントID」列を全件スキャン
    ▼
予約レコードが見つかった場合 → buildBookingCard_(booking)
予約レコードなし             → 「予約管理対象外」メッセージカード
```

### 9.4 UIカード構成

```
┌──────────────────────────────────┐
│  予約情報                         │
│  顧客名: ○○様                    │
│  メニュー: ○○                    │
│  日時: yyyy/MM/dd HH:mm          │
│  現在のステータス: 予約            │
│                                  │
│  [来店確認]    [キャンセル]       │
└──────────────────────────────────┘
```

### 9.5 ステータス更新（handleStatusUpdate_）

予約シートの該当行のステータス列を書き換え。既存の `Code.js` のステータス更新ロジック（カレンダー色変更・顧客マスタ更新）は呼ばず、スプレッドシートへの直接書き込みで実施（アドオンのスコープで完結させるため）。

### 9.6 appsscript.json 設定

```json
"addOns": {
  "common": { "name": "予約管理アドオン", "logoUrl": "..." },
  "calendar": {
    "homepageTrigger": { "runFunction": "onCalendarHome" },
    "eventOpenTrigger": { "runFunction": "onCalendarEventOpen" }
  }
},
"urlFetchWhitelist": [
  "https://api.line.me/",
  "https://notify-api.line.me/"
]
```

**追加OAuthスコープ**:
- `https://www.googleapis.com/auth/calendar.addons.execute`
- `https://www.googleapis.com/auth/calendar.addons.current.event.read`

### 9.7 モバイル対応方針

| 環境 | 方法 | 備考 |
|---|---|---|
| PCブラウザ | カレンダーアドオン（テストデプロイ） | 利用可能 |
| スマホアプリ | **kanri.html をブラウザでブックマーク** | アドオンはスマホ非対応（テストデプロイ制限） |
| スマホアプリ（将来） | Workspace Marketplace 公開後はアドオンが利用可能 | 要審査 |
