## 1. システム概要

### 1.1 アーキテクチャ概要

```
[エンドユーザー]
    │ LINEアプリ
    ▼
[LIFF アプリ]  ←── GitHub Pages ホスティング
    │ liff.html / liff.js / liff.css
    │ URL: ?gasApiUrl=<GAS_URL>&liffId=<LIFF_ID>
    │
    │ HTTPS POST (JSON + idToken)
    ▼
[GAS Web App]  ←── Google Apps Script デプロイ
    │ Code.js / calendar-addon.js / settings.html / kanri.html / unauthorized.html
    │ executeAs: USER_DEPLOYING / access: ANYONE_ANONYMOUS
    │
    ├── Google Sheets（データベース）
    ├── Google Calendar（予約イベント・ブロック予定）
    └── LINE Messaging API（Webhook受信・プッシュ通知）

[店舗スタッフ（PC）]
    │ Googleアカウントでアクセス
    ├── Googleカレンダー → カレンダーアドオン（来店確認・キャンセル）
    └── GAS Web App doGet
            ├── ?page=kanri  → kanri.html（予約管理画面）
            └── デフォルト   → settings.html（システム設定画面）

[店舗スタッフ（スマホ）]
    │ ブラウザでシステムURLをブックマーク
    └── kanri.html（予約管理画面）

[新規店舗オーナー]
    │
    ▼
[Cloudflare Worker]  ←── onboarding + デプロイ支援コンソール
    │ Google OAuth 2.0（オンボーディング）
    │ /admin: Cloudflare Access + Cookie セッション（開発者のみ）
    └── 自動プロビジョニング / コード配信 / 顧客台帳
```

### 1.2 ホスティング構成

| コンポーネント | ホスティング先 | URL形式 |
|---|---|---|
| LIFFアプリ（予約画面） | GitHub Pages | `https://[org].github.io/[repo]/liff.html` |
| GAS バックエンド | Google Apps Script | `https://script.google.com/macros/s/[ID]/exec` |
| オンボーディングサーバー | Cloudflare Workers | `https://[name].[subdomain].workers.dev` |

**整合性ルール（v3.3）**:
- `LINE User ID` は論理上1顧客1行。重複行が存在する場合、`mergeAllDuplicateCustomers_` / `mergeDuplicateCustomerRowsByUserId_` により自動統合する（管理画面一覧読み込み時・登録/更新後）。

- 管理画面の `updateCustomerInfo` は同一 User ID に一致する**すべての行**を更新する。

管理画面では予約データから以下を自動算出して表示する（シート列には保存しない）:
来店回数、初回来店日、キャンセル率、無断率、よく使うメニュー、担当者指名傾向。

## 3. バックエンド API設計（Code.js）

### 3.1 doGet エンドポイント

```
GET [GAS_URL]
GET [GAS_URL]?page=kanri
GET [GAS_URL]?action=downloadICS&bookingIds=BK001,BK002
```

**認証**: `Session.getActiveUser().getEmail()` を `adminEmail` および `staffs[].email` と照合。未認証時は `unauthorized.html` を返す。

**ICSダウンロード（GAS）**: 認証不要（URLを知っている全員がアクセス可能）。指定された予約IDのICSファイルを返す。VALARM（1時間前通知）を含む。**LIFF（iOS）からは使用しない**（§4.4 参照）。

### 3.2 doPost エンドポイント（LIFF API）

**認証（v3.1）**: 予約系 action は `idToken`（LIFF `getIDToken()`）を GAS が LINE Verify API で検証し、リクエストの `lineUserId` と一致することを要求。詳細は [セキュリティ設計書.md §6](./セキュリティ設計書.md)。

**リクエスト形式**:
```json
{"action": "アクション名", "idToken": "...", "lineUserId": "U...", ...パラメーター}
```

**レスポンス形式**:
```json
{"success": true, "data": {}}
{"success": false, "message": "エラー内容"}
{"success": false, "code": "LINE_AUTH_FAILED", "message": "..."}
```

#### アクション一覧

| アクション | 引数 | ID Token | 説明 |
|---|---|---|---|
| `getInitData` | `lineUserId`, `idToken` | ✅ | 初期データ取得 |
| `getAvailableSlots` | `date`, `duration`, `staffEmail?` | — | 週間空き枠取得 |
| `createBooking` | `bookingData`, `idToken` | ✅ | 単一予約作成 |
| `createBulkBookings` | `bookingDataList`, `idToken` | ✅ | 一括予約作成 |
| `registerCustomer` | `lineUserId`, `customerName`, `gender`, `ageGroup`, `idToken` | ✅ | 顧客登録（`有効` 行が既存ならスキップ。`無効` 行のみ存在する場合は同じ行を `有効` に更新して再有効化。登録後に重複行を統合） |

#### getInitData レスポンス

**性能（v3.5）**: Config は `getPublicConfigs_()`（ScriptCache）から取得。顧客登録状態は `findCustomerByUserId_()` で毎回判定。LINE Verify は `verifyLineIdTokenCached_()`。

```json
{
  "shopName": "string",
  "serviceMenus": [{"name": "string", "duration": number}],
  "isStaffFeatureEnabled": boolean,
  "staffs": [{"name": "string", "email": "string"}],
  "bookingTimeUnit": number,
  "bookingLookaheadDays": number,
  "isRegistered": boolean,
  "maxBulkBookings": number,
  "allowSameDayBooking": boolean,
  "alternateBookingGuide": string
}
```

#### getAvailableSlots レスポンス

```json
{
  "yyyy-MM-dd": ["HH:mm", "HH:mm", ...],
  ...
}
```

**空き枠計算ロジック（重要）**:
- **Aルート（指名なし）**: 満席タイムテーブルから高速取得 → maxConcurrentBookings と比較
- **Bルート（指名あり）**: 予約シートから該当担当者の予約をリアルタイム計算 → 上限=1
- ブロックカレンダー同期後に `ブロック予定同期` シートを参照してブロックスロットを除外
- **定休日（v3.2）**: `holidays` に含まれる曜日は当日の空き枠をすべて除外（任意設定）
- **非稼働時間（v3.2）**: `nonOperatingHours` の時間帯に該当するスロットを除外（任意設定。昼休み等）
- **当日予約ルール**: `allowSameDayBooking === false` の場合、当日の空き枠をすべて除外。LIFF 日時選択画面には `alternateBookingGuide`（未設定時はデフォルト文言）を表示
- **性能（v3.5）**: 読み取り API では満席TTの全件再構築・ブロック同期の毎回実行を行わない。詳細は **§12**

#### createBooking / createBulkBookings

**一括予約の特記事項**:
- `LockService.getScriptLock()` で排他制御（タイムアウト30秒）
- 全スロットを先行チェック（Step1）→ 全件OK後に書き込み（Step2）
- バッチ内の予約同士の重複も検証
- 1件でも空きがなければ全件キャンセル（アトミック）

### 3.3 LINE Webhook

`doPost` で `postData.events` が存在する場合にWebhook処理。

| ユーザー送信テキスト | 応答内容 |
|---|---|
| `予約` | 今後の予約一覧（日時・メニュー・担当者） |

### 3.4 バックグラウンド処理（トリガー）

| 関数名 | 実行タイミング | 処理内容 |
|---|---|---|
| `runNightlyBatch` | 毎日 2:00 | 満席タイムテーブル全件再計算 |
| `syncBlockCalendars` | 毎時 | ブロックカレンダー差分同期（Sync Token方式） |

## 4. フロントエンド設計（LIFF）

### 4.1 画面遷移

```
[起動] LIFF初期化 → LINE認証 → getInitData
    │
    ▼
Step1: ご予約について（予約経験選択）
    ├── 初めてのご予約 + 未登録 → 顧客情報登録画面
    │       └── 登録完了 → Step2（トースト「お客様情報の登録が完了しました！」）
    ├── 初めてのご予約 + 登録済み → 登録済み案内 → Step2
    └── 2回目以降のご予約
            ├── 未登録 → ボタン disabled（遷移不可）
            └── 登録済み → Step2
    │
    ▼
Step2: 予約種別選択（新規予約 / 確認・変更）
    ├── 確認・変更 → 未来予約一覧 → 1件選択 → 詳細（キャンセル / 日時変更）
    │       └── 変更時: カレンダーで新日時選択 → 確認 → rescheduleBookingByUser
    │       └── キャンセル時: 確認 → cancelBookingByUser（ICSモード時は手動削除案内）
    │
    ▼
Step3: メニュー選択
    │
    ▼
Step4: 担当者選択 → カレンダー表示 → 日時選択
    │ ※担当者未選択時はカレンダー非表示
    │
    ▼
予約確定ボタン → createBooking / createBulkBookings
    │
    ▼
完了画面（ICS でカレンダーに追加）
```

**Step1 詳細（v3.3）**:
- 見出し: 「ご予約について」。選択肢: 「初めてのご予約」「2回目以降のご予約」
- `getInitData.isRegistered === false` のとき「2回目以降のご予約」は `disabled`（登録画面へは誘導しない）
- 「初めてのご予約」+ 登録済み → 「すでにお客様情報を登録済みです」案内画面（`section-customer-already-registered`）
- 顧客登録 API 成功後は登録完了画面を経由せず Step2 へ直接遷移し、画面上部にトースト通知を表示

### 4.2 URL パラメーター

LIFFアプリのURLには以下のクエリパラメーターが必要。sessionStorageにキャッシュされる。

```
?gasApiUrl=[GAS WebアプリURL]&liffId=[LIFF ID]
```

**認証**: URL を知っていても LINE ログイン（`liff.login()`）が必要。API 呼び出し時は `idToken` を付与し GAS で検証（`lineLoginChannelId` 設定時）。リッチメニュー経由でなくとも LIFF URL から予約可能なため、URL の公開範囲は最小限にすること。

### 4.3 一括予約UI仕様

- `maxBulkBookings` が 1 の場合: 一括選択UI非表示（単一予約のみ）
- `maxBulkBookings` が 2以上の場合: カウンターチップ表示
  - 未選択時: 「カレンダーから日時を選んでください（最大N件）」
  - 選択中: 「選択済み: X / N 件（タップで確認）▼」
  - 上限到達: 未選択スロットを `bulk-limit-reached`（グレーアウト）
  - 時間帯重複: `slot-conflict`（警告色）で選択不可

### 4.4 カレンダーへの追加（ICS）

予約完了後、LIFF 側で ICS 文字列を生成し `sessionStorage` に保持する。ICS には VALARM（1時間前通知）を含む。

#### iOS LIFF（`liff.isInClient() === true`）

1. 「📅 カレンダーに追加」タップ
2. `liff.openWindow({ url: Worker ICS URL, external: true })` で Safari を開く
3. Safari が端末標準の「カレンダーに追加」UI を表示

**Worker ICS URL**（Google アカウント非依存）:

```
GET https://reservation-onboarding.reservation-onboarding.workers.dev/ics?d=<base64(UTF-8 ICS)>
```

- Cloudflare Worker が `Content-Type: text/calendar` を **inline** で返却
- **`script.google.com` の GAS URL は使用しない** — Safari の Google ログイン状態により、デプロイユーザー以外では Google ドライブエラーになるため

#### LIFF 外（PC ブラウザ等、`isInClient === false`）

- `sessionStorage` の ICS を Blob ダウンロード（`.ics` ファイル保存）

#### GAS `downloadICS` エンドポイント

- `GET [GAS_URL]?action=downloadICS&bookingIds=...` は e2e テスト・直接 URL アクセス用に維持
- LIFF 本番フローでは使用しない

## 5. 管理画面設計

### 5.1 認証

`Session.getActiveUser().getEmail()` を以下と照合（大文字小文字無視）:
- `adminEmail`
- `staffs[].email`

**初回セットアップモード（v3.0追加）**:
- `adminEmail` が未設定の場合（オンボーディング直後など）、認証チェックをスキップしてアクセスを許可する
- `doGet`（画面表示）と `saveConfigs`（設定保存）の両方に適用
- `adminEmail` が設定済みの場合は通常の認証チェックに戻る
- これによりオンボーディング後に設定画面への初回アクセスが確実に可能となる

**saveConfigs（v3.2変更）**:
- Config シート A列に存在するキーは B列を更新
- **存在しないキー（例: `lineLoginChannelId`, `nonOperatingHours`）は新規行として追加**（v3.2）
- settings.html から保存した値が Config シートに確実に反映される

## 6. ブロックカレンダー同期設計

### 6.1 同期戦略（Sync Token方式）

1. 初回（Sync Token なし）: フル同期（今日以降の全イベント）
2. 以降（Sync Token あり）: 差分同期（変更分のみ）
3. Sync Token 失効（410 Gone）: 自動でフル同期に切り替え

### 6.2 カレンダー選択ロジック

| 条件 | 使用カレンダー |
|---|---|
| 担当者機能OFF | `block_input_calendar_id`（店舗共有） |
| 担当者機能ON + 指名あり | `block_input_calendar_id`（店舗共有）＋ `staff_block_calendar_id_map[staffEmail]`（担当者個別） |
| 担当者機能ON + 指名なし | `block_input_calendar_id`（店舗共有） |

定期同期（1時間ごとのトリガー）では、上記に加え `getAllBlockCalendarIds_` により Config に登録された全ブロックカレンダーを一括同期する。

### 6.3 終日イベントの処理

終日イベントは該当日の営業時間全スロットをブロック。

## 8. オンボーディング設計（Cloudflare Worker）

### 8.1 処理フロー

```
GET /           → ランディングページ
GET /start      → Google OAuth 開始（state をKVに保存）
GET /callback   → OAuth認証完了 → プロビジョニング実行
GET /complete   → 完了ページ（URL表示）
GET /admin/login  → 管理パスワード入力（Cookie 発行）
GET /admin        → デプロイ支援コンソール
GET /ics          → LIFF 向け ICS 配信（text/calendar inline）
```

### 8.2 プロビジョニング手順

1. OAuthトークンから店舗オーナーのメールアドレスを取得（`id_token` JWTデコード → 失敗時は `userinfo` API）
2. テンプレートスプレッドシートをコピー（`Drive API files.copy`）
3. ConfigシートのreservationSheetIdに新SSのIDを書き込み（`Sheets API`）
4. **ConfigシートのadminEmailに店舗オーナーのメールを書き込み**（`Sheets API`）← v3.0追加
5. 新規スタンドアロンGASプロジェクト作成（`Apps Script API projects.create`）
6. GitHubからスクリプトファイルを取得し、`PLACEHOLDER_SPREADSHEET_ID` を実IDに置換
7. GASプロジェクトへコンテンツをプッシュ（`projects.updateContent`）
8. バージョン作成（`projects.versions.create`）
9. Webアプリデプロイ（`projects.deployments.create`）

**ヘルパー関数**:
- `getEmailFromIdToken(idToken)` : JWT のペイロードをデコードしてメール取得（主）
- `getUserEmail(accessToken)` : userinfo API でメール取得（フォールバック）
- `writeConfigValue(ssId, auth, key, value)` : Configシートの任意キーへ値を書き込む汎用関数

### 8.3 OAuthスコープ（v3.0変更）

| スコープ | 変更 | 用途 |
|---|---|---|
| `drive.file` | **drive から変更**（v3.0） | テンプレートSSコピー（アプリ作成ファイルのみ）|
| `spreadsheets` | 変更なし | Configシートへの読み書き |
| `script.projects` | 変更なし | GASプロジェクト作成 |
| `script.deployments` | 変更なし | Webアプリデプロイ |
| `email` / `profile` | 変更なし | ユーザーメールアドレス取得 |

> **変更理由**: `drive`（全ドライブアクセス）はGoogleのCASA審査（有料）が必要。
> `drive.file`（アプリ作成ファイルのみ）に限定することでCASA不要・標準審査（無料）のみで対応可能。

### 8.4 環境変数（Cloudflare Secrets）

| 変数名 | 説明 |
|---|---|
| GOOGLE_CLIENT_ID | OAuth 2.0 クライアントID |
| GOOGLE_CLIENT_SECRET | OAuth 2.0 クライアントシークレット |
| TEMPLATE_SS_ID | テンプレートスプレッドシートID |
| WORKER_URL | WorkerのURL |
| GITHUB_REPO | GitHubリポジトリ（org/repo形式） |
| GITHUB_BRANCH | ブランチ名 |
| ADMIN_SECRET_KEY | 管理ページアクセスキー |

### 8.5 KV namespace（ONBOARDING_KV）

| キーパターン | TTL | 内容 |
|---|---|---|
| `state:{uuid}` | 600秒 | OAuth state検証用 |
| `store:{ssId}` | 3年 | 導入台帳（ssId, ssUrl, onboardedAt） |

### 8.6 デプロイ支援コンソール（/admin）

**認証（v3.1）**: Cloudflare Access（Gmail）+ HttpOnly Cookie セッション（`ADMIN_SECRET_KEY`）。旧 `?key=` URL は廃止。

**コード配信フロー**:
1. 開発者がローカルで修正 → `git add / commit / push origin main`（GitHub が正本）
2. LIFF（liff.js 等）は GitHub Pages に自動反映（1〜2分）

> **注意**: コンソールは GitHub から**読み取るのみ**。ローカル未 push の変更は配信されない。
> コンソール左パネルに git コマンド（コピー用）を表示（v3.2）。

**Worker 自体の更新**: `cloudflare-worker/` 変更時は `npx wrangler deploy` が別途必要。

## 9. カレンダーアドオン設計（v3.0追加）

### 9.1 概要

Googleカレンダーのサイドパネルに表示されるアドオン。カレンダーイベントを開いたときに、対応する予約情報を表示し、ステータス変更（来店確認・キャンセル）をその場で実行できる。

- **ファイル**: `calendar-addon.js`
- **デプロイ形式**: Google Workspace アドオン（テストデプロイ）
- **対象ユーザー**: 店舗スタッフ（PC Googleカレンダー）

### 9.2 エントリーポイント

| 関数 | トリガー | 説明 |
|---|---|---|
| `onCalendarHome(e)` | アドオンホーム表示時 | 使い方案内カードを表示 |
| `onCalendarEventOpen(e)` | カレンダーイベントを開いたとき | 予約情報カードを表示 |

> **注意**: アドオンはイベントを「クリック」しただけでは起動しない。タイトルをクリックして詳細画面を開いた場合のみ `onCalendarEventOpen` が呼ばれる（Google API仕様）。

### 9.3 予約情報取得フロー

```
カレンダーイベントを開く
    │
    ▼
onCalendarEventOpen(e)
    │ e.calendar.calendarEventId を取得
    ▼
findBookingByEventId_(calendarEventId)
    │ 予約シートの「イベントID」列を全件スキャン
    ▼
予約レコードが見つかった場合 → buildBookingCard_(booking)
予約レコードなし             → 「予約管理対象外」メッセージカード
```

### 9.4 UIカード構成

```
┌──────────────────────────────────┐
│  予約情報                         │
│  顧客名: ○○様                    │
│  メニュー: ○○                    │
│  日時: yyyy/MM/dd HH:mm          │
│  現在のステータス: 予約            │
│                                  │
│  [来店確認]    [キャンセル]       │
└──────────────────────────────────┘
```

### 9.5 ステータス更新（handleStatusUpdate_）

予約シートの該当行のステータス列を書き換え。既存の `Code.js` のステータス更新ロジック（カレンダー色変更・顧客マスタ更新）は呼ばず、スプレッドシートへの直接書き込みで実施（アドオンのスコープで完結させるため）。

### 9.6 appsscript.json 設定

```json
"addOns": {
  "common": { "name": "予約管理アドオン", "logoUrl": "..." },
  "calendar": {
    "homepageTrigger": { "runFunction": "onCalendarHome" },
    "eventOpenTrigger": { "runFunction": "onCalendarEventOpen" }
  }
},
"urlFetchWhitelist": [
  "https://api.line.me/",
  "https://notify-api.line.me/"

```

**追加OAuthスコープ**:
- `https://www.googleapis.com/auth/calendar.addons.execute`

### 12.2 現状の問題（v3.5 改修前）

| 症状 | 主因（Code.js） |
|---|---|
| LIFF 起動が遅い | GAS コールドスタート + Config シート毎回読み + LINE Verify API 毎回 + 顧客マスタ走査 |
| 空き枠表示が遅い | **`getAvailableSlots` 内で毎回 `rebuildTimetableFromReservations_()`**（予約全件読み→満席TT全書き換え）+ **`syncBlockCalendars()` 毎回** |
| 予約確定が遅い | 今回は対象外（カレンダー同期作成は維持） |

### 12.3 `getAvailableSlots` 読み取り最適化

#### 12.3.1 基本方針

- **読み取り API は読み取り専用**とする。満席タイムテーブルの全件再構築・ブロックカレンダーの差分同期は **書き込み経路と定期バッチ** に限定する。
- **正（ソース・オブ・トゥルース）**: 予約シート。満席タイムテーブルは派生キャッシュ（§2.1.3）。

#### 12.3.2 Aルート（指名なし）の変更

| 項目 | 改修前 | 改修後（v3.5） |
|---|---|---|
| 満席TT | 毎回 `rebuildTimetableFromReservations_()` 後に読み取り | **既存シートを読み取りのみ**（週表示範囲でフィルタ） |
| 整合性 | 毎回再構築で強制同期 | 予約作成/変更/取消・夜間バッチで更新 |

**満席TT を更新するタイミング（書き込み経路）**:

| トリガー | 関数 | 更新回数 |
|---|---|---|
| 単一予約作成 | `createBooking` → `rebuildTimetableFromReservations_()` | **1回** |
| 一括予約作成 | `createBulkBookings` → 全件書き込み完了後 **`rebuildTimetableFromReservations_()` 1回のみ** | **1回**（各件ループ内では更新しない） |
| ユーザー取消・変更 | `cancelBookingByUser` / `rescheduleBookingByUser` → `rebuildTimetableFromReservations_()` | 1回 |
| 管理画面ステータス変更 | `confirmBookingStatus` → `releaseTimetableSlots_()`（内部1回再構築） | 1回 |
| 夜間バッチ | `runNightlyBatch`（満席TT全件再計算） | 1回 |

> **v3.5.1 修正**: 旧記載「各件 `updateTimetableSlots_`」は現行実装の欠陥（N件でN回フル再構築）をそのまま写していたため廃止。`updateTimetableSlots_` は v3.5 実装では **`rebuildTimetableFromReservations_()` のエイリアス（1回再構築）** として整理する。

#### 12.3.3 ブロックカレンダー同期の間引き

| 項目 | 改修前 | 改修後（v3.5） |
|---|---|---|
| `syncBlockCalendars` | `getAvailableSlots` **毎リクエスト** | **最短間隔（5分）以内はスキップ** |
| 根拠 | — | `PropertiesService` に `blockSyncLastRunAt`（ISO8601）を記録 |
| 定期同期 | 毎時トリガー | **変更なし**（毎時 `syncBlockCalendars`） |

スキップ時も `ブロック予定同期` シートの **既存データ** を参照してブロックスロットを計算する。最大5分のブロック反映遅延を許容する。

#### 12.3.4 Bルート（指名あり・予約変更時）

- 引き続き予約シートからリアルタイム計算。
- **週表示範囲（7日）かつステータス「予約」「仮予約」** に絞って走査し、全件読みのメモリ負荷を抑える（§12.5.1）。

#### 12.3.5 スキーマ検証（fail-fast）

API 入口（`getAvailableSlots` / `createBooking` 系）で、対象シートの **必須ヘッダー存在** を検証する。

| シート | 必須列（例） |
|---|---|
| 予約 | 予約ID, ステータス, 予約日時, 終了日時, 担当者Email |
| 満席タイムテーブル | 日付, 時間枠, 予約数 |
| ブロック予定同期 | 予定ID, カレンダーID, 開始日時, 終了日時 |

欠落時は API エラーを返し、エラーログシートに記録。**列の手動挿入・削除によるサイレント破損を早期検知**する（§12.5.2）。

### 12.4 CacheService（ScriptCache）設計

#### 12.4.1 定数

| 定数名 | 値 | 用途 |
|---|---|---|
| `CONFIG_CACHE_TTL_SEC` | 300（5分） | Config キャッシュの上限 TTL |
| `LINE_VERIFY_CACHE_MAX_TTL_SEC` | 600（10分） | LINE Verify 結果キャッシュの上限 TTL |
| `LINE_VERIFY_CACHE_SKEW_SEC` | 30 | exp 切れ前の安全マージン |
| `BLOCK_SYNC_MIN_INTERVAL_SEC` | 300（5分） | ブロック同期の最短間隔 |

#### 12.4.2 Config キャッシュ

**方針**: Config 全体を1キーに載せない（100KB 制限・秘密情報混入を避ける）。**LIFF/予約 API 向けサブセット**のみキャッシュする。

**キャッシュキー**: `config:public:v1`

**キャッシュ値（JSON）に含めるキー**:

```
shopName, serviceMenus, businessHours, holidays, nonOperatingHours,
isStaffFeatureEnabled, staffs, bookingTimeUnit, bookingLookaheadDays,
maxConcurrentBookings, maxBulkBookings, allowSameDayBooking,
alternateBookingGuide, reminderMode, lineLoginChannelId,
reservationCalendarId, reservationSheetId
```

**含めないキー**（毎回シート読み取り、または別経路）:

```
lineChannelAccessToken, lineChannelSecret, adminEmail,
staff_block_calendar_id_map, block_input_calendar_id, archiveSpreadsheetId,
serviceSuspended  ※停止チェックはキャッシュ外で最新を参照
```

**実装関数（予定）**:

| 関数 | 役割 |
|---|---|
| `getPublicConfigs_()` | ScriptCache → ミス時 `getConfigs()` からサブセット抽出 → put |
| `invalidateScriptCaches_()` | 全キャッシュキー削除 |

**サイズ上限**: put 前に `JSON.stringify` 長を計測。**90KB 超なら put せず**毎回シート読み（100KB 制限の安全マージン）。

#### 12.4.3 LINE ID Token 検証キャッシュ

**方針**: Verify API の結果（`sub`）を **LINE User ID（sub）単位** でキャッシュする。JWT の `exp` と `clientId` を必ず考慮する。

**キャッシュキー**: `lineVerify:v1:{sub}`（例: `lineVerify:v1:U1234abcd...`）

> **v3.5.1 修正**: idToken ハッシュ単位の動的キーは ScriptCache 500キー上限と矛盾するため廃止。**1ユーザー1キー** とする。

**キャッシュ値（JSON）**: `{ "sub": "U...", "exp": 1730000000, "clientId": "1234567890" }`

**キャッシュヒット時の検証**（いずれか不一致ならキャッシュ破棄 → Verify API 再呼び出し）:

- `clientId` === 現在の `getPublicConfigs_().lineLoginChannelId`
- `exp` > 現在時刻 + `LINE_VERIFY_CACHE_SKEW_SEC`
- 呼び出し側 `expectedUserId` === `sub`

**TTL 計算**（put 時）:

```
ttlSec = min(LINE_VERIFY_CACHE_MAX_TTL_SEC, exp - nowSec - LINE_VERIFY_CACHE_SKEW_SEC)
```

`ttlSec <= 0` の場合はキャッシュせず Verify API を呼ぶ。

**実装関数（予定）**:

| 関数 | 役割 |
|---|---|
| `verifyLineIdTokenCached_(idToken, configs)` | sub キーでヒット判定 → sub 返却。ミス時 Verify API → exp/clientId 付きで put |
| `requireVerifiedLineUserId_` | 内部で `getPublicConfigs_()` + `verifyLineIdTokenCached_()` を使用 |

#### 12.4.4 顧客登録状態（getInitData）

**方針**: ユーザー依存のため Config と同様に長期キャッシュしない。

- `findCustomerByUserId_()` は **毎回シート参照**（現状維持）。
- 将来、短 TTL（60秒）の per-user キャッシュを検討可。v3.5 初版では **実装しない**（整合性優先）。

#### 12.4.5 キャッシュ無効化（purge）トリガー

| イベント | 処理 |
|---|---|
| `saveConfigs` 成功 | `invalidateScriptCaches_()` |
| （将来）Worker が Config を直接更新 | GAS 側 purge API または TTL 5分で自然失効 |

**手動編集**: スプレッドシート上で Config を直接変更した場合、最大5分間古い設定が残る。**運用**: 変更後は settings から再保存するか、5分待つ。

#### 12.4.6 doPost の Config 読み取り統合

`doPost` は **LIFF API** と **LINE Webhook** の共通入口（§3.2 / §3.3）。経路ごとに Config の取り方を分ける。

| 経路 | 判定 | Config の取り方 |
|---|---|---|
| LINE Webhook | `postData.events` あり | **`getConfigs()` フル読み（キャッシュ対象外）**。`lineChannelAccessToken` 等の秘密情報が必要 |
| LIFF API | `postData.action` あり | 下記フロー |
| 自動テスト | `action === 'runTests'` | 従来どおり（停止チェックもスキップ） |

**LIFF API フロー（改修後）**:

1. `getConfigValue_('serviceSuspended')` 等で **停止フラグのみキャッシュ外読み**（Config 全行パースを避ける）
2. 以降の action 処理は **`getPublicConfigs_()` を1リクエスト1回**取得して引き回す（`requireVerifiedLineUserId_` 内の二重読みを廃止）

**doPost 対象外（現状維持）**:

| 経路 | Config |
|---|---|
| `doGet`（settings / kanri テンプレート埋め込み） | `getConfigs()` |
| `google.script.run`（管理画面 saveConfigs 等） | `getConfigs()` / `saveConfigs` 内 |

**purge 対象**: `saveConfigs` 成功時は **`config:public:v1` のみ削除**。`lineVerify:v1:{sub}` は TTL + ヒット時 `clientId` 照合で自然失効（§12.4.3）。

### 12.5 リスクと緩和

#### 12.5.1 OOM（データ増加）

| リスク | 緩和 |
|---|---|
| 予約・ブロック行数増加で `getValues()` が肥大化 | Bルートは **7日範囲 + アクティブステータス** に絞る。満席TTは週範囲フィルタ後に Map 化 |
| 満席TT再構築を読み取りから除去 | 書き込み経路 + `runNightlyBatch` で整合。§7 アーカイブで過去TT削除 |

#### 12.5.2 列挿入・削除による即死

| リスク | 緩和 |
|---|---|
| 手動列操作で `indexOf` がずれる | §12.3.5 スキーマ検証で **fail-fast** |
| 再構築頻度低下で気づきが遅れる | 夜間バッチ + 管理画面への「最終再構築時刻」表示（復旧設計で拡張予定） |

#### 12.5.3 キャッシュ呪い（更新漏れ）

| リスク | 緩和 |
|---|---|
| 設定変更が LIFF に反映されない | `saveConfigs` → 即 purge。TTL 5分の上限 |
| Worker 経由 Config 更新 | v3.5 では TTL 自然失効。復旧設計で purge 連携を検討 |

#### 12.5.4 ID Token `exp` すり抜け

| リスク | 緩和 |
|---|---|
| 期限切れトークンが通る | TTL = min(上限, exp - now - 30秒)。exp 切れはキャッシュしない |

#### 12.5.5 ScriptCache 100KB 制限

| リスク | 緩和 |
|---|---|
| Config 丸ごと put で失敗 | **public サブセットのみ** + 90KB 超えたら put しない |
| 500キー上限 | **固定キー** `config:public:v1`（1本）+ **`lineVerify:v1:{sub}`**（アクティブユーザー数程度）。idToken ハッシュキーは使わない。上限超過時は LRU evict され Verify が再実行されるだけ（機能劣化、障害ではない） |

#### 12.5.6 ブロック同期5分スキップ

| リスク | 緩和 |
|---|---|
| 直前に追加したブロックが空き枠に未反映 | 毎時定期同期 + 最大5分遅延を許容。緊急時は GAS 手動 `syncBlockCalendars`（復旧設計で Runbook 化） |

#### 12.5.7 満席TT読み取りのみ化

| リスク | 緩和 |
|---|---|
| TT と予約シートの一時的不整合 | 予約 **書き込み直後** に `updateTimetableSlots_`。夜間 `runNightlyBatch` で全件再計算 |
