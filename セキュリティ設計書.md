## 1. 目的とスコープ

本書は、本システムを構成する複数環境（GitHub Pages / GAS / Cloudflare Worker / LINE / Google）を横断する**認証・認可・秘密情報管理**を定義する。

| 対象 | 含む | 含まない |
|------|------|----------|
| デプロイ支援コンソール | ✅ | — |
| LIFF 予約 API | ✅ | — |
| 店舗管理画面（settings / kanri） | ✅ | — |
| オンボーディング | ✅ | — |
| LINE Webhook | 一部（署名検証は既存） | Messaging API 全般 |
| 顧客データの暗号化 at rest | — | Google インフラに依存 |

## 2. 脅威モデル

| ID | 脅威 | 影響 | 対策状況 |
|----|------|------|----------|
| T-01 | 管理コンソール URL + 旧 `?key=` 漏洩 | 全店舗 GAS 改ざん・停止 | ✅ 二段構え（Access + Cookie） |
| T-02 | LIFF URL（gasApiUrl + liffId）漏洩 | 任意 LINE ユーザーによる予約 | ⚠️ LINE ログイン必須。✅ ID Token 検証で POST なりすまし防止 |
| T-03 | GAS `/exec` への直接 POST | 偽 lineUserId で予約 | ✅ ID Token 検証（lineLoginChannelId 設定時） |
| T-04 | Webhook 偽装 | 不正イベント処理 | ✅ Channel Secret 署名（既存） |
| T-05 | 設定画面への不正アクセス | Config 改ざん | ✅ Google アカウント許可リスト（既存） |
| T-06 | OAuth オンボーディング悪用 | 不正店舗作成 | ⚠️ 公開 URL（ビジネス要件）。state + KV TTL |

## 4. 認証・認可マトリクス

| コンポーネント | URL 例 | 認証方式 | 認可 |
|----------------|--------|----------|------|
| LIFF 予約 | `*.github.io/.../liff.html` | LINE LIFF ログイン + **ID Token（GAS 検証）** | ログイン済み LINE ユーザーのみ予約可 |
| GAS doPost | `script.google.com/.../exec` | action ごと（下表） | — |
| GAS settings/kanri | 同上 `?page=kanri` | Google セッション | adminEmail + staffs |
| Worker オンボーディング | `workers.dev/` | Google OAuth | スコープ同意ユーザー |
| Worker `/admin` | `workers.dev/admin` | **Access Gmail + Cookie** | 開発者のみ |
| LINE Webhook | GAS `/exec` POST | 署名検証 | Channel Secret |

### 4.1 GAS doPost action 別

| action | ID Token 検証 | 備考 |
|--------|---------------|------|
| getInitData | ✅ 必須（channel 設定時） | lineUserId 一致確認 |
| registerCustomer | ✅ 必須 | |
| createBooking | ✅ 必須 | bookingData.lineUserId と一致 |
| createBulkBookings | ✅ 必須 | 全件同一 lineUserId |
| getAvailableSlots | ❌ 不要 | 空き枠参照のみ（将来レート制限可） |

## 5. 管理系セキュリティ（二段構え）

### 5.1 第1層: Cloudflare Access

- 対象パス: `/admin`, `/admin/*`
- Identity: Google（`tomimori.systems4150@gmail.com`）
- ヘッダー: `Cf-Access-Authenticated-User-Email`
- 設定手順: [docs/Cloudflare_Access_設定手順.txt](./docs/Cloudflare_Access_設定手順.txt)

### 5.2 第2層: HttpOnly セッション Cookie

| 項目 | 値 |
|------|-----|
| Cookie 名 | `admin_session` |
| Path | `/admin` |
| 属性 | HttpOnly, SameSite=Strict, Secure（HTTPS 時） |
| 有効期間 | 8 時間 |
| 署名 | HMAC-SHA256（`ADMIN_SECRET_KEY`） |
| ログイン | `POST /admin/login`（パスワードは URL に載せない） |
| ログアウト | `GET /admin/logout` |

### 5.3 旧 URL 移行

- `?key=ADMIN_SECRET_KEY` 付き URL は初回アクセス時 Cookie に変換してリダイレクト
- ブックマークは `/admin` のみに統一

## 6. LIFF 予約系セキュリティ（ID Token 検証）

### 6.1 背景

改修前は GAS がリクエスト JSON の `lineUserId` をそのまま信頼していた。LIFF URL 漏洩 + 直接 POST でなりすまし可能だった。

### 6.2 方式

```
[LIFF] liff.init → liff.getIDToken()
         │
         │ POST { action, idToken, lineUserId, ... }
         ▼
[GAS]  verifyLineIdToken_()
         │  LINE API: POST /oauth2/v2.1/verify
         │  id_token + client_id (= lineLoginChannelId)
         ▼
       返却 sub === リクエスト lineUserId ?
         Yes → 処理続行
         No  → 401 相当 JSON エラー
```

### 6.3 Config キー

| キー | 用途 | 取得場所 |
|------|------|----------|
| `liffChannelId` | LIFF アプリ ID（URL の `liffId` パラメータ） | LINE Developers › LIFF |
| `lineLoginChannelId` | ID Token 検証用 Channel ID | LINE Developers › 該当チャネル › Basic settings › Channel ID |

※ `liffChannelId` は名称上「チャネル」だが実態は **LIFF ID**。検証には数値の **LINE Login Channel ID** を `lineLoginChannelId` に保存する。

### 6.4 エラーレスポンス

```json
{
  "success": false,
  "code": "LINE_AUTH_FAILED",
  "message": "LINE認証の検証に失敗しました。LINEアプリから再度お試しください。"
}
```
