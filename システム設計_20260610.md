# システム要件定義・仕様一覧

## 前提
| セクション/モジュール | 項目名 | ステップ詳細内容 / アクション | 備考 / システムロジック |
| :--- | :--- | :--- | :--- |
| A. ターゲット | 対象業態 | パーソナルジム・個人経営美容系サロン・美容室・整骨院など小規模リピートサービス | 想定予約数は最大50人/日 |
| B. 運用想定 | デプロイ方式 | **ヘッドレスアーキテクチャと共通基盤ライブラリを組み合わせたハイブリッド方式。** | **フロントエンド：GitHub Pagesに共通基盤として公開。バックエンド：主要ロジックは開発者管理の「共通基盤ライブラリ」に集約し、各顧客環境のGASからはそのライブラリを参照するのみ。** |
| B. 運用想定 | 操作IFの原則 | **エンドユーザが操作するのは「LIFF（GitHub Pages）」と「Googleカレンダー」。店舗スタッフが操作するのは「店舗向け管理Webアプリ（GAS）」と「Googleカレンダー」のみ。** | スプレッドシートは直接編集・閲覧しない（バックグラウンド動作） |
| C. システム概要 | 基本アーキテクチャ | **ヘッドレスアーキテクチャ。フロントエンド（LIFF/静的HTML）をGitHub Pages、バックエンド（API/データ処理）をGASで構築。両者はHTTPS(fetch API)で非同期通信。店舗向け管理画面は従来通りGASのWebアプリで提供。** | エラーログ出力UI実装・LINEからLIFFへの遷移で予約。 |

## 決定事項
| セクション/モジュール | 項目名 | ステップ詳細内容 / アクション | 備考 / システムロジック |
| :--- | :--- | :--- | :--- |
| システム導入方法 | サポート方針 | デプロイと初期設定は必ず顧客自身のGoogleアカウントで実行。オンライン導入サポート（画面共有ガイド）/対面導入/セルフ導入から選択。 | |
| 事業継続計画（BCP） | データ保護オプション | 管理者アカウント失念時の損失回避。プランA：顧客管理の予備アカウントに編集者権限付与。プランB：開発者アカウントを緊急時復旧用に設定。 | プランBは顧客データアクセス/インシデント等のリスク同意・書面同意が必須 |
| アカウント運用 | デプロイ・担当者変更方針 | 離脱リスク低減のため「店舗共有の管理用Googleアカウント」や「オーナーアカウント」での構築を強く推奨。原則：ログイン権限の継承（アカウント譲渡）。例外：継承不可時は編集権限を持つ別アカウントで再デプロイ。 | |
| 技術的制約と対策 | URL固定とライブラリ保守 | **顧客側のコード修正や再デプロイ（＝URL変更）を原則行わないため、主要ロジックを開発者管理の「共通基盤ライブラリ」に集約。顧客側GASはライブラリ参照のみ。機能修正はライブラリ更新で一斉反映。** | **LIFFのURL（GitHub Pages）と店舗向け管理画面のURL（GAS）は固定され、顧客負担なし。** |
| 技術的制約と対策 | バージョン管理と自動更新 | **ライブラリ参照時は「開発モード」禁止、特定の安定バージョン（v1, v2等）を指定。カスタムメニュー「[▼システム管理] > [システムを最新版に更新する]」による半自動アップデート。** | |
| 技術的制約と対策 | APIへのアクセス認証 | **不正利用防止。エンドユーザ向けAPI：LIFFから取得したIDトークンをGAS側で検証。店舗向け管理画面：GAS標準の`Session.getActiveUser().getEmail()`でGoogleアカウントを検証し、正規の担当者かチェック。** | 管理者引き継ぎ時は速やかに開発者へ通知しリスト更新が必要（更新まで一時利用不可）。 |
| 店舗個別設定 | 予約可能期間・昼休み・休み区分 | 予約可能期間：当日〜最大3ヶ月後。昼休み：has_fixed_break=TRUE時、指定時間帯は全担当者共通で予約不可。休み区分区分設定は廃止（カレンダー予定はすべて枠外）。 | |

## カレンダー設定・同期ロジック
| 分類 | セクション/モジュール | 項目名 | 詳細内容 / システムロジック |
| :--- | :--- | :--- | :--- |
| カレンダー役割 | IDと役割の固定 | カレンダーID設定 | reservation_calendar_id（予約カレンダーID） / block_input_calendar_id（ブロックカレンダーID:OFFのみ） / staff_block_calendar_id_map（ONのみ） |
| 担当者機能OFF | 店として1つの枠運用 | 運用ルール | 予約カレンダー（店舗共有：システムが作成・表示） / ブロックカレンダー（別の共有：予定入力用）。予約不可にしたい時間はブロックカレンダーに予定入力。**※予約カレンダーにブロック目的の予定を入力しない（二重控除回避）** |
| 担当者機能ON | 同時予約あり得る | 運用ルール | 予約カレンダー（店舗共有：システムが作成） / 担当者別ブロックカレンダー（staff_block_calendar_id_mapで1人1つ固定し予定入力用）。**※店舗共有カレンダーの予定は予約枠外抽出対象に含めない（二重控除回避）** |
| 共通 | 空き枠算出 | 基本ロジック | 空き枠 ＝ 営業時間 － 既存予約（予約シート）－ 予約枠外区間（ブロック予定同期シート） |
| Googleカレンダー予定同期 | Sync Token運用 | 公式仕様整合 | 反映は差分同期（Sync Token）で行いsyncTokenはカレンダーIDごと保存。失効（410 Gone）検知時はシート全消去→フル同期再実行。**※nextPageToken発生時はページングを最後まで追う** |
| Googleカレンダー予定同期 | 予定確認タイミング | 予約処理開始トリガー | LIFFを開いたタイミングで最新化。(A)Sync Tokenによる即時増分同期（推奨） (B)期間指定での都度参照（補助：計算用でシートに永続反映しないも可）**※定期同期のタイムラグによる「直近で入れた予定が反映されない」リスクを低減** |

## データベース設計（スプレッドシート構造）

データストアとして機能する「予約管理システム」スプレッドシートの各シートのヘッダー構成（スキーマ）は以下の通り定義する。

### 1. 予約 シート
エンドユーザーからの予約情報を管理し、Googleカレンダーと連携するメインシート。
* **カラム構成:** `['予約ID', 'ステータス', 'LINE User ID', '顧客名', 'メニュー名', '予約日時', '終了日時', 'イベントID', '担当者名', '担当者Email']`

### 2. 満席タイムテーブル シート
店舗全体の空き枠計算を高速化するためのキャッシュ用シート（夜間バッチ処理等で更新）。
* **カラム構成:** `['日付', '時間枠', '予約数']`

### 3. Config シート
システム全体の設定値（メニュー、営業時間、スタッフ情報、同時予約上限など）を管理するシート。
* **カラム構成:** A列に `Key`（設定項目名）、B列に `Value`（設定値またはJSON文字列）を配置する。
* **主要Keyの追加定義:** `maxBulkBookings`（整数、デフォルト1）：1回の予約操作で選択できる日時枠の上限数。

### 4. 顧客マスタ シート（※未実装・今後追加要件）
初回ユーザーの顧客情報登録フローにて使用するシート。
* **カラム構成:** 現在未定義（※顧客登録画面の実装フェーズにて、必要な個人情報に合わせて定義を追加すること）。

### 5. SaaS運用およびUX強化要件
SaaSとしての提供形態を確立するため、以下の機能要件を追加する。

* **初期導入の完全自動化（OAuth活用）**
  * **仕様:** 顧客による手動デプロイを廃止。Google Cloud APIおよびOAuth 2.0を活用した専用のオンボーディング用サーバー（またはフロー）を構築する。ホスティング先は **Cloudflare Workers** を推奨（無料枠で運用可能）。
  * **実装時期:** フェーズ4完了後に着手。
  * **目的:** 顧客はログインと権限承認の数クリックのみでシステム導入を完了させ、手動作業による設定ミスや導入ハードルを排除するため。

* **アーカイブデータの別ファイル退避**
  * **仕様:** メインDB（予約シート等）の行数増加に伴うGASの実行速度低下を防止するため、一定期間経過した古い予約データを、定期的なバッチ処理により「アーカイブ用別スプレッドシートファイル」へ自動エクスポート（およびメインDBからの削除）する仕様を導入する。
  * **目的:** システムのパフォーマンスを長期間維持するため。

* **顧客カルテUIの実装**
  * **仕様:** 管理画面上に、スプレッドシートの生データとは別に、整形された「顧客カルテ・来店履歴一覧」を表示するUIを実装する。
  * **目的:** 非IT層のユーザーに対しても、蓄積されたデータの価値を直感的に体感させ、システムの継続利用率を高めるため。
